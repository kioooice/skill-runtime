# Development Workflow

这份文档定义一套可复用的轻量开发流程，目标只有一个：减少白干、减少范围漂移、减少无效确认，让新工具和新应用都能按同一套方式推进。

## 适用范围

适合下面这类工作：

- 从 0 到 1 做一个新工具
- 为某个新想法判断是否值得开发
- 进入 MVP 设计和实现
- 连续迭代一个还在验证阶段的应用

不适合：

- 已经成熟的大型正式产品流程
- 需要重审批、重文档、重协作管理的企业项目

## 核心原则

- 先判断值不值得做，再判断怎么做。
- 先做最小闭环，不做缩小版完整产品。
- 先给出可试结果，再继续加东西。
- 用户主要判断目标、体验和结果，不默认审代码。
- 没有真实分叉、风险升级或试用节点时，默认连续推进。

## 两种模式

开发时先分清当前处于哪一种模式。

### 1. 探索模式

用于判断问题是否成立、有没有必要开发、替代方案是否已经够用。

这一阶段要做的事：

- 澄清问题是什么
- 判断是不是高频真实问题
- 检查现有工具或流程能不能解决大部分需求
- 用手工方式做低成本验证
- 决定是否进入 MVP

不要在探索模式里直接开始实现。

### 2. 执行模式

用于问题已经相对清楚，开始做 MVP 或推进某一轮开发。

这一阶段要做的事：

- 只围绕一个主目标推进
- 做出一个最小可试结果
- 用试用反馈决定下一轮

不要在执行模式里重新打开过多立项讨论。

## 新工具前置流程

当一个新工具想法出现时，按下面顺序判断。

### 第 1 步：先定义问题

至少先说清：

- 想做什么
- 为什么会冒出这个念头
- 当前最不爽的点是什么

如果想法还很模糊，先做问题澄清，不进入开发。

### 第 2 步：判断有没有必要

重点看这几个问题：

- 这个问题是不是反复出现
- 它是不是持续困扰
- 它是不是明显浪费时间、精力或质量
- 不解决它，是否还会继续反复打断工作

如果只是偶尔想到、偶尔不爽，先不要开发。

### 第 3 步：检查替代方案

强制先看现有工具、现有流程或手工方法：

- 现有方案能不能解决 70% 到 80%
- 问题到底是工具缺失，还是习惯没有建立
- 为什么不是继续优化现有方案

如果现有方案已经基本够用，优先不开发。

### 第 4 步：做手工验证

在写代码前，先用最低成本模拟真实流程，比如：

- 文档
- 表格
- Notion
- Obsidian
- 其他现成工具

手工验证要观察：

- 实际会不会用
- 最麻烦的是哪一步
- 最有价值的是哪一步
- 用完后是否还想继续

如果手工版都不愿意持续用，通常不值得立刻产品化。

### 第 5 步：进入 MVP

只有下面三件事相对清楚时，才进入 MVP：

- 要验证的核心假设是什么
- 最小闭环是什么
- 这轮明确不做什么

## MVP 默认做法

MVP 不是缩小版完整产品，而是最小可验证闭环。

### MVP 的目标

只验证最关键的问题，例如：

- 用户会不会真的用
- 主路径能不能走通
- 结果是否有价值
- 当前流程最痛的阻塞点是什么

### MVP 的默认顺序

1. 定一句话目标
2. 定唯一主路径
3. 列出必须做
4. 列出明确不做
5. 实现主路径
6. 给出可试结果
7. 根据反馈进入下一轮

### MVP 停止点

开发前先定义停点。通常到下面这个程度就该停下来试：

- 用户能完成一次核心任务
- 数据或结果真的形成闭环
- 已经能暴露真实问题
- 再往下做主要是在增加完整感，而不是增加验证价值

## 连续开发流程

进入执行模式后，默认按下面方式推进。

### 每一轮只做一件事

每轮只围绕一个主目标推进，例如：

- 先打通录入
- 先打通保存
- 先打通搜索
- 先打通某个关键页面

不要一轮里同时铺太多支线。

### 每一轮都尽量有可试结果

一轮结束后，优先给用户一个可以实际操作的结果，例如：

- 现在可以新建
- 现在可以保存
- 现在可以搜索
- 现在可以走通某条流程

没有可试结果的开发轮次要尽量少。

### 默认连续推进

如果没有下面这些情况，不要反复停下来问“要不要继续”：

- 出现真实分叉，需要用户选方向
- 风险明显升级，继续做可能浪费很多时间
- 已经到了该让用户亲自试用的节点

其他情况下，默认继续推进，并简短汇报：

- 刚完成了什么
- 现在能试什么
- 是否需要用户做决定
- 如果不需要，下一步会直接做什么

## 用户反馈方式

默认不要求用户审代码。更适合的反馈方式是：

- 我试了什么
- 哪一步顺
- 哪一步卡
- 结果是不是我要的
- 目前最难受的一点是什么

这样的反馈比“继续”更有价值，也更适合非工程师主导的开发协作。

## 技能和插件使用规则

默认不要把“找技能”和“研究插件”当成前置工作。

只有下面情况，才值得主动考虑扩展：

- 明显属于专门工作流
- 明显需要外部平台集成
- 明显属于重复高摩擦操作

如果值不值得用技能或插件不明确，优先由助手判断，不把筛选成本转嫁给用户。

## 默认提问格式

当想法还不确定时，先用最短输入启动即可：

- 我想做：
- 我冒出这个念头，是因为：
- 我现在最不爽的是：

当问题已经相对清楚时，再补：

- 现有替代方案：
- 替代方案为什么不够：
- 这次只验证什么：
- 做到哪里先停：

## 最终目标

这套流程最终要达成的不是更规范，而是更省力：

- 少做不值得做的东西
- 少在错误方向上深挖
- 少做无意义的“继续确认”
- 让每一轮开发都更接近真实验证
